## Where combat intents live
- `src/engine/combatExchange.js` is the whole atomic-exchange engine. `COMPANION_ACTIONS = new Set(['attack', 'defend', 'pass'])` (as of 2026-07-17) — normalized in `normalizeCombatExchange`.
- Resolution order per exchange (`planCombatExchange`): enemy_condition_updates → player slots (`resolvePlayerSlots`) → companions (`resolveCompanions`) → `enemy.defending = false` reset → enemies (`resolveEnemies`). Companions always act before enemies within one exchange.
- `defend` (companion): sets `companion.defending = true`, gives up its attack. Consumed in `resolveEnemyAttack` via `targetDisadvantage = !!companion.defending` — i.e. defend gives the ATTACKER disadvantage against that companion (self-preservation), it does not protect anyone else.
- Enemy attack targeting: `resolveEnemyAttack({ enemy, targetRef, ... })` in combatExchange.js line ~1177. `targetRef !== 'player'` looks up a companion by ref; otherwise defaults to the player. Missing/invalid companion target intent silently drops that actor's slot (never redirects to player) — established existing pattern worth mirroring for any new redirect feature.
- Uncanny Dodge (Rogue L5+) is strictly scoped to `targetType === 'player'` and is a single shared `uncannyDodgeState` object across one whole turn (including the multi-actor Opening Initiative loop) — see line ~1217. Any new redirect mechanic MUST perform the redirect (player ref → companion id) before this targetType check is evaluated, or Uncanny Dodge will incorrectly consume/trigger on a hit that was never a real player hit.
- Companion AC lookup already composes `(companion.ac ?? 10) + (companion.spellAcBonus || 0)` (line ~1194) — sustained buffs like Shield of Faith cast on a companion already flow into any target-AC calculation for free; no extra plumbing needed for a tanking/guard feature.
- Companions have NO incapacitation gate today: `resolveCompanions` never checks stunned/paralyzed/unconscious before letting a companion act (unlike enemies, which are blocked via `getIncapacitatingCondition` in `resolveEnemies`). This is a pre-existing asymmetry — flag it if any new companion action (e.g. `guard`) needs "can't declare this while incapacitated" semantics.
- Opening Initiative (`planOpeningExchange`, phase `opening`): actors who win initiative resolve via `resolveCompanions({ exchange: null, ... })` / `resolveEnemies({ exchange: null, ... })` — there are NO `companionIntents` yet at this point (the player hasn't submitted a `combat_exchange`). So `defend` (and any future `guard`) cannot apply during the opening slot; it only takes effect from the first normal exchange onward. This is expected/consistent, not a bug.
- Low-level solo safety (`terminalState`, line ~1324): `lowLevelSolo = level <= 2 && !party.some(isCompanionActive)`. Recruiting ANY living companion — guard-capable or not — immediately removes the soft-defeat safety net and exposes the level 1-2 hero to real dying/death-save rules. This is a meaningful, currently-undocumented-to-players tradeoff worth calling out whenever a "protect the hero" companion feature is discussed, since such a feature is exactly what makes that tradeoff worth taking.
- Prompt-side targeting guidance lives in `src/llm/promptBuilder.js` lines ~419-441 (COMBAT NOTES block). As of 2026-07-17 there is no targeting-discipline guidance — the example enemy intent shows `"target":"player"` and missing targets default to player (line 154 of combatExchange.js: `target: ref(intent.target) || 'player'`), so the DM has no nudge away from always-hits-the-hero. Companion example stats for a "guard" role companion: `hp:18, maxHp:18, ac:14, attackBonus:4, damage:"1d8+2"` (line 348) — treat this as the reference statline for balance math on any tanking-adjacent companion mechanic (enemy attack bonus range is +3..+6 typically, `DEFAULT_ENEMY_ATTACK_BONUS=3`, bounds enforced in `enemyStats.js`: attack bonus clamps to [-5,15], AC clamps to [1,25]).

## 2026-07-17 design review: DM targeting discipline + companion `guard` intent
Reviewed a proposal (not yet implemented) to (1) add DM prompt guidance for enemies to choose targets from fiction instead of defaulting to the player, and (2) add a `guard` companion intent that fully redirects player-targeted attacks to the guardian for the exchange (first living guardian in party order soaks; falls through to player if the guardian drops mid-exchange).

Verdicts given:
- Full redirection (no advantage-vs-guardian, no AC penalty) is the right call. Guardian statline (AC14, 18 HP) vs enemy +3..+6/1d6+2..2d6+3 already yields real risk (45-80% hit chance, dies in 2-4 hits), and losing the attack action is a genuine DPS tax — no extra rider needed. Adding attacker advantage vs the guardian would make the tactic self-defeating and discourage its use entirely.
- Guard should NOT also grant defend's attacker-disadvantage. Keep the two mutually exclusive: `defend` = self-preservation (reduces the odds YOU get hit), `guard` = altruism (guarantees the PLAYER doesn't get hit, at your own normal odds). Stacking disadvantage onto guard makes `defend` strictly worse for companions in every case, deleting its niche.
- No extra numeric guardrail needed against "perma-guard trivializes fights" beyond the existing cost (lost attack + real risk of being downed) — PROVIDED companion guard-role statlines stay tuned so a guardian drops within a normal fight (roughly the 15-22 HP / AC 13-15 band already used in the example). This is a tuning discipline, not a new mechanic.
- Implementation must scope the redirect substitution to before Uncanny Dodge's `targetType === 'player'` check (see above) and should add an incapacitation gate on declaring/honoring `guard` (a stunned/paralyzed/unconscious companion shouldn't be able to newly guard) since no such gate exists for companions today.
- Change 1's targeting-fiction language ("front-line companion screens a back-line caster") risks narrating protection that isn't mechanically real for companion-to-companion attacks, since guard v1 (as scoped) only redirects attacks aimed at `player`, not at other companions. Recommend explicit prompt wording distinguishing narrative target selection (flavor only) from the formal `guard` mechanic (the only thing that actually redirects), and requiring enemies to have a genuine fictional reason (engaged in melee, dealt them damage, visibly exposed) before targeting a companion — never a metagame "attack the squishiest HP total" choice, and no repeated pile-on finishing off an already-downed companion without real stakes.
