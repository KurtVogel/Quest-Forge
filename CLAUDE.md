# CLAUDE.md

Guidance for Claude Code (and humans) working in this repo. Read this first — it's the fast path to being productive here.

## What Quest Forge is

A single-player, browser-based tabletop RPG where an **LLM plays the Dungeon Master** and a **client-side engine owns all the hard mechanics** — dice, rules math, character state, and persistence. There is no backend of its own: the browser calls Gemini/OpenAI directly and (optionally) syncs saves to a *user-supplied* Firebase.

## North star

**Make the most out of the LLM. That is the money-maker; the engine enables the magic.**
Use the engine to make mechanics reliable, fast, and inspectable, then spend the LLM where it
matters most: vivid narration, consequence, memory, NPC intent, atmosphere, and turning raw
state changes into lived fiction. A correct number is not enough if the moment should feel
alive. LLM-rich does not mean verbose: ordinary turns should be brief enough to keep the
player in motion.

- **Stack:** React 19 + Vite 7, mostly plain `.js`/`.jsx` (one `.ts`: the dice engine). Package name is `rpg-client`.
- **Deploy target:** Firebase Hosting, project `quest-forge-99ab1` → https://quest-forge-99ab1.web.app
- **Tests:** vitest (`npm test`) covers the rules math, the death-save state machine in the reducer, roll resolution with mocked dice, and golden-fixture cases for `responseParser` LLM quirks. Run it before committing engine/parser changes; add a fixture whenever a new LLM failure mode is discovered.

## The one idea that explains everything: the DM↔engine contract

The LLM **narrates and declares game events; it never rolls dice or mutates state.** The client rolls every die (cryptographically random) and is the single source of truth for state. Each turn:

1. Player submits an action — **`ChatPanel.jsx`** is the orchestrator.
2. **`promptBuilder.js`** builds the system prompt from live state (character, inventory, quests, world facts, combat, retrieved memories) plus strict DM pacing rules.
3. The DM streams back **narrative + a trailing ` ```json ` event block**.
4. **`responseParser.js`** splits narrative from JSON and normalizes it. It is defensively coded against LLM misbehavior: unfenced/malformed JSON → repair; roll-requests-written-in-prose → text detector; an outcome narrated *before* a roll → flagged for correction.
5. Outside combat, **`rollResolver.js`** resolves `requested_rolls` with real dice (`dice.ts`) and auto-sends results back for narration. Active combat never uses that roll path: the DM emits bounded `combat_exchange` intent, `combatExchange.js` validates live targets/actions and generates every player/companion/enemy roll, and `APPLY_COMBAT_EXCHANGE` commits the complete result atomically by `exchangeId`. A separate narration-only call reads the stored result; it cannot reroll or mutate mechanics and can be retried after failure/reload.
6. **`applyEvents()`** dispatches each change into the reducer.

**Implication when editing:** mechanics belong in the engine, not the prompt. If the DM "should" do math (HP, XP, AC, to-hit, leveling), the client already does it — the prompt explicitly tells the DM *not* to. Adding a feature usually means: a JSON field the DM emits → a normalize step in `responseParser.js` → a reducer action → (optionally) a prompt instruction telling the DM the field exists.

## Subsystems worth knowing

- **State:** one `useReducer` in `gameReducer.js`; all mutations go through dispatched actions. `LOAD_GAME` deliberately keeps the live `user` and merges current `settings` over the save — don't let old saves clobber auth/settings (older saves have stale/missing values). Both save paths (local IndexedDB and cloud Firestore) persist the SAME snapshot via `serializeGameState()` in `persistence.js` — spread + strip `user`/`ui`/secrets, stamped with `saveVersion`. **Never reintroduce a field whitelist there**: that's how `fronts` and `pendingRoleplayCheck` silently vanished from local saves until 2026-07-03. `LOAD_GAME` heals front-less established campaigns (pre-fix saves) with a deterministic front reseed and reopens the Settings Dynamic-World upgrade. Autosave failures surface as an error toast, and a `visibilitychange`/`pagehide` flush protects the last turn from the 2s debounce window.
- **Layered memory** (keeps long campaigns inside the token budget while making callbacks feel intentional):
  1. **World facts** — canonical truths, never compressed.
  2. **Journal** — every ~10 messages, Gemini Flash summarizes and prunes older messages from the LLM window (`worldJournal.js`).
  3. **Story memory** — compact narrative-only callback cards (`storyMemory`) for promises, wounds, player canon, mysteries, relationship beats, foreshadowing, and NPC agendas. `storyMemory.js` scores cards by relevance/location/NPC/salience/cooldown and injects only a few as `## DRAMATIC CALLBACK OPPORTUNITIES`; the DM may mark a used/resolved card with `memory_updates`, but this never changes mechanics.
  4. **RAG** — world facts / journal / NPCs / story-memory cards / player messages embedded with Gemini `gemini-embedding-2` at 768 dimensions. Stored memories use Google's asymmetric retrieval-document format and scene context uses its matching search-query format; vectors are schema-tagged in IndexedDB and retrieved by cosine similarity (`vectorMemory.js`). **Gemini-only.**
  Plus the **Scribe** (`scribe.js`): a silent Gemini-Flash pass after each turn that extracts world facts / NPC updates / story-memory cards **and character/NPC visual appearance** (`appearance` on NPCs, `player_appearance` on the character — fed to scene art AND to the DM's own prompt: `## KNOWN NPCs` carries `looks:`, the hero's appearance sits in the PLAYER CHARACTER block, and NPC RAG embeddings include looks, so established features like white hair stay consistent in prose across long campaigns). Appearance capture is **exempt from the extraction budget**, and every Scribe call receives the KNOWN APPEARANCES for the characters in the turn so appearance updates must emit the complete merged description — a new scar can never replace-and-erase the rest of the look (600-char clamp at the reducer boundary). Appearance capture is **shame-free** (DECISIONS.md 2026-07-05): body proportions and intimate/unflattering details the fiction establishes are canon like any scar — the Scribe must record them frankly and merges may never launder the record; the DM prompt likewise forbids quietly tidying up an established body. Since 2026-07-15 (DECISIONS.md) records are **frank in content, clinical in register**: the whole Scribe family (extraction, reflection, NPC enrichment, art director) writes durable records in neutral anatomical vocabulary at full specificity — never crude slang, which was tripping Gemini API safety guidelines when re-injected into every machinery call; merges restate old crude wording neutrally so records self-clean, and narration explicitness stays a player choice via the custom DM prompt. **Player↔NPC relationships get the same treatment** (DECISIONS.md 2026-07-05): the Scribe (and optionally the DM) emits `stanceToPlayer` — the NPC's complete personal stance toward the hero, merged against KNOWN PLAYER-RELATIONSHIP STANCES exactly like appearance, and since 2026-07-09 backstopped engine-side (DECISIONS.md): `mergeNpcDossierText` in `npcRoster.js` token-containment-merges the durable dossier fields `personality`/`goals`/`secrets`/`stanceToPlayer` in the reducer — a per-turn fragment appends, a restatement drops, only a complete rewrite replaces, and the oldest sentences fall off first at the 600-char cap — and `callbackHooks` is a capped rolling shortlist (`appendCallbackHooks`, cap 5), so the immediate scene can never erase an NPC's record; `appearance` keeps its prompt-contract replace (haircuts/disguises must be able to drop details) and `lastNotes`/`agenda`/`relationshipTension`/`privateNotes` stay current-state by design — plus per-turn `bondMoment` beats that the reducer appends into a capped, deduped, append-only `bondMoments` history; both are exempt from the budget, injected into `## KNOWN NPCs` (`toward the hero:` / `personal history with the hero:`), embedded in NPC RAG text, weighted in prompt curation, and shown on the character card as "Toward you" + "Moments between you" (the card also surfaces the `appearance` field as a "Looks" block — captured looks including body proportions were feeding the DM + scene art but were previously invisible in the Journal). Journal "Deepen memory" backfills stance, bond moments **and a merged physical appearance** for existing campaigns by reading the recent chat messages that mention the NPC (verbatim conversations the journal otherwise prunes); pre-stance records re-flag as Thin. Extraction is otherwise **budgeted**: the prompt demands ≤2 facts + ≤2 cards per ordinary turn (zero on most), the engine caps both at 3 per turn (reflection cards at 2), and `ADD_WORLD_FACT(S)` reject near-duplicate restatements via stopword-stripped token containment — over-extraction was the top 2026-07-03 playtest finding. Story-memory cards get the same treatment (DECISIONS.md 2026-07-14): `findStoryMemoryMatch` containment-merges same-type reworded restatements of one beat into the existing card (possessives folded), and on merge a fragment can never clobber a richer text — the first keyed memory eval found one promise recorded 4× under reworded subjects. On the journal cadence it also runs a cheap NPC/front reflection pass for agenda, relationship pressure, front symptoms, and future callback hooks. The same per-turn pass doubles as the **loot persistence audit**: it compares the final narrative against the loot events actually applied and grants only the missing shortfall (coins/items the DM narrated but never emitted), with the HERO'S CURRENT INVENTORY in context so using/drawing/lighting owned gear is never re-granted as an acquisition (2026-07-09 Grok finding; the DM ECONOMY prompt carries the matching inverse rule), deduped per narration message — or per exchangeId on victory narration, whose narration-only contract otherwise discards all DM events — via `CLAIM_LOOT_SOURCE`, clamped by the engine, and announced with a visible system line. No regex fallback: if the Scribe is unavailable the audit skips (DECISIONS.md 2026-07-02). It owns `composeScenePrompt`, the on-demand art-director call that builds the image prompt from the current situation + accumulated appearances when the player requests scene art. The DM itself only ever sees a 20-message sliding window (`MESSAGE_WINDOW` in `ChatPanel.jsx`).
  Full design note: `docs/LLM_WOW_LAYER.md`.
  - **Campaign premise** (`session.premise`): the player's opening scenario, captured at adventure start ("Set the stage" field), with an 8,000-character allowance for substantial faction, geography, history, conflict, and character scaffolding. Injected as a never-pruned `## CAMPAIGN PREMISE` block (`buildPremiseBlock`, DM rule 8) — the one path for *player-authored* canon, which the journal otherwise compresses away as non-event setup. On a fresh campaign, character creation sets the one-time `session.openingScenePending` marker and ChatPanel **auto-opens the first scene from the premise**. That same one-time opening reconciles concrete portable items explicitly established as the hero's possessions against live starting inventory via bounded `starting_items`; the engine normalizes catalog mechanics, rejects exact/catalog duplicates, equips only when the premise says the item begins worn/wielded, and excludes other entities' or merely mentioned objects. Leave the premise blank to keep the classic manual start. Continue/Load never calls the DM automatically: it restores the exact conversational handoff and waits for the player.
- **Player narrative authority:** the player owns their character's intent, words, inner life, and harmless compatible color—not external reality or automatic outcomes. The DM prompt welcomes emergent comedy/absurdity but treats unsupported advantage, danger-erasure, or world rewrites as wishes/attempts and responds from established fiction without scolding. Raw player RAG entries are labeled non-canonical statements/attempts, and the Scribe records external player claims only when the DM narrative accepts them; compatible personal backstory/vows remain valid player canon.
- **OOC table talk** (`llm/tableTalk.js`, DECISIONS.md 2026-07-09): messages addressing the DM out of character (`OOC:`, `(OOC)`, `[ooc]`, `/ooc`, or `DM,`/`GM:`/`Dungeon Master:` at message start) are detected deterministically in ChatPanel and answered under a dedicated `## CURRENT RESPONSE MODE — OUT-OF-CHARACTER TABLE TALK` block; a standing DM rule covers unprefixed meta questions best-effort. A table-talk turn pauses the world: it never enters the combat-intent machine (in combat the phase stays `awaiting_player`), parsed events are force-nulled so a disobedient DM cannot mutate state or request rolls, and the exchange is kept out of memory (no player/narrative RAG embeds, no Scribe extraction). The DM recaps and takes tone/pacing requests freely but never reveals hidden state (front titles/clocks, secret NPC motives, private notes). Never rely on provider goodwill for OOC — Gemini breaks character graciously on its own; Grok does not.
- **NPC naming diversity:** new DM/front-created people and person-like factions must avoid the shared stock-name list in `nameGuidance.js` (Elara/Elora/Elyra, Silas/Sylas, Thorne/Thorn, Kael, Lyra, Rowan, Aria, Liora, Seraphina, Nyx, Zephyr, Aldric, Garrick, Mira, Vesper, Cassian). Names should emerge from culture/community with varied phonetics and levels of ornament. This never renames premise/world-fact/NPC/journal/memory/player-established people.
- **Out-of-combat check discipline:** dice are exceptional, not the default response to roleplay. A check requires genuine uncertainty, active opposition/pressure, and an interesting failure consequence; routine competence and approaches that already solve the obstacle succeed without a roll. Clever positioning earns automatic success, advantage, or a lower DC. Outside-combat `requested_rolls` carry `advantage`/`disadvantage`; `rollResolver.js` rolls two d20s, keeps high/low, combines condition effects, and shows the result. The DM adjudicates whether fiction warrants it; the engine owns execution. The solo-play ladder is 8 easy under pressure / 10 standard / 12 meaningful / 15 strong opposition / 18+ exceptional—never default DC 15. One roll settles the immediate approach: one proportionate consequence, then a meaningful choice—not repeated checks for the same objective or a punishment cascade. Failed checks control external outcomes, not the player's authored delivery or inner state. `outOfCombatRollPolicy.js` uses the background Scribe model (with local regex/keyword rules as an offline fallback) to semantically audit checks, rejecting (a) belief/innocence checks when the player explicitly answers truthfully without seeking a concrete concession and (b) checks whose sole outcome is whether the hero maintains an authored calm/stoic/brave/composed/sincere demeanor. The Scribe also semantically detects and extracts text-based check requests when the DM narrates rolls in prose instead of JSON, eliminating brittle regex keyword scanners. ChatPanel hides the invalid setup and requests a no-roll response. NPC doubt and external consequences remain free; release/access/aid/risk may warrant a roll, and genuine saves against spells, poison, supernatural fear, or defined physical effects remain valid.
- **Roleplay check proposals:** outside combat, `requested_rolls` never roll immediately. They become a reload-safe `pendingRoleplayCheck` with a public table-facing adjudication (`reason`, `opposition`, `failure_stakes`, `difficulty_reason`, and situational advantage/disadvantage reason). The player chooses **Roll**, **Challenge ruling** once, or **Change approach** before any dice exist. A challenge makes the DM withdraw, revise, or uphold the proposal with concise public justification; an upheld/revised ruling is final, preventing endless negotiation and outcome-based reroll bargaining. Accepted and follow-up out-of-combat rolls use the same proposal boundary. Loot the DM (incorrectly) attaches to a roll proposal is never granted client-side: it rides the proposal as metadata and returns to the DM as a grant-or-deny reminder in the post-roll outcome and challenge prompts — dice can deny it, and the Scribe loot audit remains the backstop when a narrated grant lacks events. Withheld setup narration is never erased: it rides the proposal (`setupNarrative`/`setupMessageId`), gets re-injected into the post-roll outcome prompt so the DM re-establishes its fiction (the hidden setup is otherwise invisible to both player and DM history), and **Change approach** reveals the hidden message with a marker (`REVEAL_MESSAGE`) instead of discarding it — except when it pre-narrated an outcome. Checks the Scribe extracts from natural prose (no JSON) keep their narration visible and stage the proposal beneath it; only JSON-declared or outcome-spoiling setups are withheld. Rulings that end **without dice** are durable table history: `recentRulings` (reducer-owned, cap 5, expires after ~24 messages or a location change) records withdrawn and set-aside proposals and injects a binding `## RECENT TABLE RULINGS` prompt block — withdrawn means the approach succeeds diceless on retry, an ordinary set-aside retried gets the identical check back (never reworded/re-priced), and a set-aside upheld ruling stays final with the challenge spent. Active combat bypasses this completely and remains the fast atomic exchange machine.
- **Hidden fronts & world tempo** (`fronts`, `worldTempo`, `locations`, `recentEncounters` — DECISIONS.md 2026-07-14 ×2): private campaign clocks that are never shown to the player — and since the world-tempo system, **never shown to the DM as a dossier either** (hiding beats instructing; the old always-visible `## HIDDEN CAMPAIGN FRONTS` block made every campaign violent by ~turn 7). `engine/worldTempo.js` renders a compact `## WORLD TEMPO — PRIVATE PACING STATE` block instead: the pace setpoint (`settings.paceDial`: slow-burn/standard/breakneck, Settings → Game) vs engine-measured recent heat (deterministic, from live combat/recent fights/hero wounds) with a thermostat guidance line; one-line front stubs (id + faction only — the DM keeps its `front_updates` channel); and at most ONE permitted symptom card when the current tempo directive's window is open, otherwise an explicit QUIET instruction (quiet scenes are complete scenes; player-sought danger — "I go hunt goblins" — is NEVER gated). The journal-cadence reflection proposes the `tempo_directive` (which front may surface, where, max intensity); the engine validates it: intensity clamps to the band the front's clock/stage justifies (whispers → indirect → presence → confrontation), theater gating clamps to whispers away from a front's known home (theaters grow organically — placing a directive's symptom somewhere records that place as the front's territory in the location registry), same-front and slow-burn alternation force quiet cadences, replays dedupe by cadenceId, and an **engine-rolled crypto timing die** delays the window opening by 0–4 scenes (arc reasoning decides what/where, dice decide when). The reflection may also propose ONE **emergent front** when the player has repeatedly engaged a durable uncovered threat (`normalizeEmergentFront`: complete-or-nothing, deduped, max 4 active, born at clock 0). `engine/locationRegistry.js` folds drifting DM location strings into canonical records via token containment ("Library landing, Clockwork Tower" = "Clockwork Tower") with Scribe-classified profiles (`location_profile`: haven/settlement/wilderness/frontier/hostile_site + intrinsic danger — a place's own nature, separate from imported front pressure). `recentEncounters` (appended by END_COMBAT, cap 6) feeds heat plus a vary-or-escalate/cleared-stays-cleared prompt line. Opening scenes establish normal life BG1-style — premise pressure at most atmosphere — unless the premise itself starts in medias res (`sessionPriming.js`). New campaigns keep a deterministic local-pressure fallback while `frontDirector.js` privately generates 2–3 premise-grounded, interacting pressures with bounded driving factions/forces, goals, stances, and cross-front relationships; generation runs on the slow DM model while play continues, so the reducer accepts a late-arriving install as long as the fallback front is still untouched (clock/stage 0 — DECISIONS.md 2026-07-14, the first keyed memory eval's race). Every successful journal cadence feeds its fresh summary/decisions/consequences into the private Scribe reflection; the LLM may propose only -1/0/+1 movement with a canonical reason and fictional symptom, while `fronts.js`/the reducer validate known IDs, reject jumps and stale/duplicate cadence batches, pace the clocks (only one front may gain clock per cadence, and never the same front on consecutive cadences; softening and symptom-only updates are never throttled), derive non-regressing portent stages, and autosave front-only changes. A cadence alone is not a reason to advance, and ordinary DM `front_updates` are reserved for immediate player interference or symptoms established in that response. The DM leaks only in-world symptoms; titles, clocks, stages, and notes remain private. When the player has no companions, fronts can introduce recruitable possibilities organically without forcing party membership; actual joining still uses `add_companions`. Established campaigns can explicitly **Upgrade This Campaign to Dynamic World v2** once from Settings → Game: `frontUpgrade.js` uses bounded premise/facts/journal/quests/NPC/story-memory/recent-event context, enriches every existing front by exact ID with faction relationships, may add only enough grounded pressures to reach 2–3 total, blocks combat-time/repeat/wrong-session commits, and atomically preserves every clock, stage, portent, hint, note, and game mechanic.
- **Combat:** `START_COMBAT` rolls initiative for the player, companions, and validated enemies. Actors who beat the player receive one engine-owned **Opening Initiative** slot (surprise adjusts who qualifies); a player action that started combat is queued behind that opening, never dropped. Combat-start enemy IDs and same-response intent references are canonicalized together before that queued action resolves. Afterwards combat is player-centered: the DM declares `player_slots`, optional companion intent, and bounded enemy intent (`attack|defend|flee|surrender`) in `combat_exchange`; it supplies no dice or HP authority. `combatExchange.js` resolves player → companion → active-enemy order from live state, so a foe overcome before its slot cannot act. Invalid/missing player targets block the whole plan; invalid NPC targets drop only that actor's slot and never silently retarget. Reducer phases are `opening`, `awaiting_player`, `awaiting_intent`, and `awaiting_narration`; mechanics commit once, narration acknowledges by matching `exchangeId`, then the reducer advances or closes combat and awards overcome-XP (death, flee, or surrender); a fight lost or escaped still pays XP for foes genuinely slain before the end (never for foes who fled/surrendered on a loss). Every persisted result carries an authoritative post-exchange player/enemy/companion snapshot; narration is explicitly bound to each combatant's alive/defeated/fled/surrendered state, contradictory Scribe facts/memories are discarded, and non-terminal combat prose is not embedded into long-term RAG. Persisted results make narration retry/reload idempotent. Standard profiles cover weapons, basic Wizard/Cleric attack spells, combat checks/saves, Dodge/Dash/Disengage/Interact/Pass, Extra Attack, death saves, companion actions, and enemy intents. Unsupported specials are rejected visibly without granting hostile actions. The Combat panel is compact by default on phones.
- **Enemy conditions:** enemy health (`healthy|bloodied|critical|dead`) is separate from bounded mechanical conditions (`prone`, `blinded`, `restrained`, etc.). `enemy_condition_updates` synchronizes conditions already established by authoritative fiction before player rolls; successful Check slots can apply a validated `on_success` condition; enemy intents can clear conditions immediately before their own action. Attacker and target conditions combine through `CONDITION_EFFECTS`, including advantage against prone foes and disadvantage on prone attackers. Conditions persist through exchanges/reloads and render on enemy cards.
- **Situational combat rulings:** table-style negotiation survives engine ownership through a bounded `situational_ruling` (`advantage|disadvantage` plus a required fictional reason) on player slots and companion/enemy attack intents. The DM adjudicates whether established fiction supports the ruling; a player claim alone is insufficient. The engine still rolls every die, combines the ruling with condition effects and normal cancellation, and exposes the reason in the system result. Only explicit flanking-style player attack rulings propagate to companions attacking the same target; generic advantage sources stay local, and companion-specific rulings are preserved.
- **Combat resources and safety:** Action Surge activates only on a living player turn, grants exactly two arbitrary action slots, and clears only inside a successful atomic exchange. Second Wind and healing potions remain tracked bonus actions; rests are rejected during combat. Character Sheet rest buttons resolve mechanics first and request one narration-only fictional beat; rests already narrated by the DM do not request a duplicate. Shared `enemyStats.js` validation covers parser/start/load/update/pre-roll boundaries; loaded 0-HP enemies remain dead and malformed saves cannot inject offensive stats or crash loading. Legacy `requested_rolls` are rejected while combat is active. Low-level solo 0-HP remains a non-lethal setback; higher-level dying characters stay in the exchange machine for engine-owned death saves.
- **Combat orchestration lock:** after the player sends a combat action, phase `awaiting_intent` prevents UI resources or a second input from changing the snapshot while the DM translates intent. A reload safely returns this uncommitted phase to `awaiting_player`; no dice or HP existed to replay.
- **Combat intent pacing:** active-combat intent translation remains separate from authoritative post-roll narration, but the first call is JSON-only and emits no speculative prose. Chat shows distinct intent/narration wait labels and logs per-call TTFT/total timing for tuning; never merge the calls in a way that narrates before engine results exist.
- **Progression:** `progression.js` owns XP thresholds (D&D 5e-style per-level increments: 300 XP for level 1 → 2, 600 for level 2 → 3, etc.), the D&D max-level cap (20), fixed average HP-on-level-up (`floor(hitDie / 2) + 1 + CON`, minimum 1), feature unlocks, and pending Ability Score Improvement state at the full 5e cadence — levels 4, 8, 12, 16, 19, uniform across classes, no feats (player spends exactly two ability points in the Character Profile; reducer recalculates CON HP and AC; pending is always derived as earned − applied, so old saves gain missed ASIs on load). Fighter has a coded combat bonus (`getLevelBonus`, capped at +3), engine-owned Fighting Styles (Defense AC, Dueling damage, Great Weapon damage rerolls, Archery attack; old Fighters default to Defense), Champion as the level-3 Martial Archetype (old level 3+ Fighters default to Champion; crits on natural 19-20), Extra Attack at L5 (each `combat_exchange` Attack slot expands to two independently targeted strikes), and Action Surge as a pending two-slot state after the UI resource is spent.
- **Loot/inventory:** `items.js` is the catalog for common weapons, armor, shields, consumables, and D&D-style prices. Purchases should use the `purchase` event so `gameReducer.js` validates funds and atomically subtracts coin/adds items; purchase and sell events are one-shot transactions — `recentPurchases`/`recentSales` ledgers suppress nearby LLM replays unless the player's own message explicitly trades again (repeat phrasing like "another"/"a few more of those" is honored, and post-roll outcome responses keep the player's action context for that check). Equipping/unequipping gear uses `equipment_changes` (`equip` / `unequip`) so narration like "I remove my armor" updates `equipped` flags and recomputes AC; use `items_lost` only when the item leaves inventory. Healing potions are Inventory-owned bonus actions: the client rolls healing, consumes one stack item, applies revival cleanup, and marks the combat bonus-action slot. Equipped slots are normalized through `equipment.js`: one active weapon, one armor, one shield, and no shield while a two-handed weapon is active. Descriptive catalog names such as "massive warhammer" may resolve by complete-name suffix, but recognized catalog mechanics override LLM fields; non-equipment can never remain equipped. Magic equipment supports `magicBonus` +1 to +3; weapon bonuses affect engine-owned attack/damage rolls, armor/shield bonuses affect computed AC.
- **Quests:** `quest_updates` statuses `new|updated|completed|failed` all round-trip — `new`/`updated` upsert by id/normalized name, `failed` uses `FAIL_QUEST` and shows in the Quests panel's finished section. The DM prompt's QUEST TRACKING INSTRUCTIONS require opening a quest whenever the hero accepts a job/deal/debt/investigation (even informal) and closing it in the same response that resolves it.
- **Character roster & hero files:** heroes (`character` + `inventory` — *not* campaigns; saves own those) live in a local roster (IndexedDB `characters` store via `persistence.js`) and travel as versioned JSON exports (`characterVault.js`, format `quest-forge-character` v1). The creation wizard forks into "Forge a New Hero" / "Use an Existing Hero"; the character sheet has Save to Roster / Export File. Imports are **untrusted input**: identity fields are validated/clamped and all derived fields (proficiency, saves, traits, features, resources, hit dice) are rebuilt from race/class data. Roster heroes start new adventures rested and keep their roster `id`, so re-saving updates their entry.
- **Scene-art quality contract:** Scene mode preserves both the setup and decisive aftermath of long narration and instructs the art director to include every supported subject/species/count/action/reaction without inventing generic party members. `imageGen.js` reports whether xAI or the lower-quality Pollinations fallback produced the image; Scene Art must label missing-key/xAI-failure fallbacks visibly, and fallback caching must never prevent a later xAI retry.
- **Config & secrets:** no `.env` **for the app** — the player enters their **LLM API key in-app** (Settings → AI Provider); it lives in `localStorage` via `persistence.js`. The optional playtest/eval scripts may read `GEMINI_API_KEY` from a git-ignored local `.env`. Cloud sync is **bring-your-own-Firebase** (Settings → Cloud Sync) — `quest-forge-99ab1` is only the hosting target, not the user-data backend. Cloud saves live at Firestore `users/{uid}/saves/{slotId}`; payloads too large for one document are split into an atomic batched `chunks` subcollection (no 1 MiB ceiling — required for "infinite" campaigns), so deployed `firestore.rules` must include the `chunks/{chunkId}` match. Firebase Hosting must serve `/` and `/index.html` with no-store/revalidation so live deployments cannot strand players on stale combat code; hashed `/assets/**` files stay immutable.
- **Default custom DM prompt:** RPG-first adult low-fantasy tone with strict player agency and roll discipline. Adult/explicit content is allowed only when it emerges naturally from scene logic and player choices, not sexualized by default; Settings → Custom DM Instructions → Reset to default restores it from `initialGameState.settings.customSystemPrompt`.
- **Providers:** Gemini (default, `gemini-3.1-pro-preview`), OpenAI, and xAI (`grok-4.3`, OpenAI-compatible chat API via `providers/xai.js`) for **DM narration**. The **Gemini machinery is provider-independent and mandatory** (`llm/machinery.js`, DECISIONS.md 2026-07-08): RAG embeddings, the Scribe, journal summaries, roll-policy audits, NPC enrichment/fodder review, and scene-prompt composition always run on Gemini Flash via `getBackgroundConfig()` / `getMachineryGeminiKey()` — when the DM is Gemini the main key doubles as the machinery key, otherwise the player supplies `settings.geminiApiKey` (Settings → AI Provider, stripped from saves), and **ChatPanel refuses to start a turn without it** (playing without memory quietly rots a campaign). Front *generation* (`frontDirector`/`frontUpgrade`/`frontMigration`) intentionally stays on the DM model — creative work, not extraction. **Scene art and character portraits use xAI (Grok Imagine)** — `imageGen.js` calls `grok-imagine-image-quality` at `api.x.ai`, using a **separate `settings.imageApiKey`** (Settings → AI Provider; stripped from saves like the chat key). Pasted xAI keys are trimmed and normalized with the required `xai-` prefix when omitted (shared `providers/xaiKey.js`, also used by the xAI chat provider), and Firebase Hosting's CSP must keep `https://api.x.ai` in `connect-src` or browser fetches fail before reaching xAI. Falls back to a free provider (Pollinations) if no xAI key or the call fails. The scene-art *prompt* is composed by the Scribe (Gemini machinery), the image is *rendered* by xAI — two different keys. Player portraits are generated from confirmed `character.appearance` text, use 3:4 `1k`, and downscale stored xAI data URLs to keep saves/hero files compact.

## Conventions & gotchas

- **Git workflow — master only (Vesa, 2026-07-14):** features are worked one at a time; no feature branches, no PRs. Start every session by pulling latest `origin/master`, commit on `master`, push to `origin master`. If a hosted/agent session is forced to start on a working branch, land the result on master anyway (`git push origin HEAD:master`); stale remote `claude/*` branches are harmless leftovers (the remote-session git proxy blocks branch deletion — prune them from the GitHub UI).
- 4-space indent, ES modules, function components + hooks. Match the surrounding file's style.
- `dice.ts` is the only TypeScript file; everything else is JS/JSX. `tsconfig.json` exists but the app is **not** type-checked in CI.
- Player-facing dice must stay crypto-random (`crypto.getRandomValues`, via `dice.ts`) — never `Math.random()`. This is the project's "the LLM can't cheat the dice" guarantee.
- `responseParser.js`, `scribe.js`, and `jsonExtractor.js` carry a lot of hard-won resilience against LLM output quirks. Edit carefully — regressions here break the game loop silently.
- `npm run lint` is expected to pass. Keep it clean when changing code.
- Windows-first repo (paths, `install.cmd`).
